PROJECT TITLE: Student-Mesh Distributed ML Trainer (The "Software GPU")

1. PROJECT DESCRIPTION
This project is a high-performance distributed systems infrastructure designed to 
parallelize Machine Learning training across a heterogeneous "mesh" of consumer 
devices (student laptops and mobile phones). It mimics the architecture of a GPU 
at the software layer by treating individual devices as "cores."

The system specifically addresses the hardware limitations of student groups by 
partitioning large datasets (e.g., computer vision datasets) and distributing 
the mathematical workload across a local WiFi network. Key features include 
asynchronous gradient aggregation to handle "stragglers" (slow devices) and 
fault-tolerant orchestration to ensure training continues if a device disconnects.

2. SYSTEM ARCHITECTURE (HIGH-LEVEL)
The system follows a Leader-Worker (Parameter Server) model:
- LEADER: Manages the global model state, task scheduling, and weight updates.
- WORKERS: Native C++ agents (Laptops) or Web-workers (Mobile) that calculate 
  gradients on local data batches.

3. MICROSERVICES
- API Gateway (FastAPI): Entry point for job submission, RBAC, and monitoring.
- Dataset Sharder: Logic for partitioning large datasets into Mini-Batches.
- Task Orchestrator: Tracks worker health (Heartbeats) and manages task lifecycle.
- Parameter Server: The core engine that performs Staleness-Aware Gradient 
  Aggregation (averaging updates based on Version IDs).
- Metrics Service: Real-time computation of Accuracy, F1-Score, and AUC.

4. DATABASE SCHEMA (PostgreSQL)
- Workers Table: [worker_id, device_type, os, compute_cap, max_mem, last_seen]
- Jobs Table: [job_id, model_type, target_acc, current_version_id, status]
- Data_Batches Table: [batch_id, job_id, storage_path, assigned_to, status, retry_count]

5. IN-MEMORY STATE (Redis)
- Global Weights Cache: Binary storage of current model tensors for fast access.
- Heartbeat Registry: TTL-based keys to monitor active device connectivity.
- Version Map: Maps worker_id to their assigned version_id for staleness calculation.

6. COMMUNICATION PROTOCOLS
- User <-> API Gateway: REST (HTTP/2) for job management.
- Leader <-> Worker (Control): gRPC for bi-directional heartbeats and tasking.
- Worker <-> Parameter Server (Data): gRPC/Protobuf for high-speed tensor transfer.
- Dashboard <-> Metrics: GraphQL for efficient, selective data fetching.

7. EXECUTION ENGINE
- High-Performance Worker: Native C++ implementation for CPU/GPU acceleration.
- Lightweight Worker: JavaScript/ONNX Runtime for cross-platform mobile support.
- Model Registry: Final serialization service to export .pt or .weights artifacts.
